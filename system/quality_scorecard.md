# VacAX Quality Scorecard v1

목적: 결과물을 매번 같은 기준으로 평가해서 "좋아졌다"를 비교 가능하게 만든다.

## 기본 점수표

작업 종료 시 0~5점으로 빠르게 채점한다.

| 항목 | 점수 | 기준 |
|---|---:|---|
| 목표 달성도 | 0-5 | 처음 목표를 실제로 달성했는가 |
| 검증 수준 | 0-5 | lint/build/test/실행/사람검수 중 적절한 검증을 했는가 |
| 사용자 개입 감소 | 0-5 | 다음번 CEO 손이 덜 가게 만들었는가 |
| 재사용성 | 0-5 | 다음 프로젝트/세션에서 다시 쓸 수 있는가 |
| 포트폴리오 가치 | 0-5 | AX/AI 기획 이직 증거로 설명 가능한가 |
| 운영문서 반영 | 0-5 | STATE/NEXT/LOG/LEARNINGS에 흡수됐는가 |

총점: 30점

## 판정

- 25~30: Golden 후보. `knowledge/goldens.md` 등재 검토
- 18~24: 사용 가능. 다음 작업에서 보완
- 10~17: 부분 성공. 실패 원인 1줄 필수
- 0~9: 재작업 또는 폐기. NEXT에 복구 액션 1개 고정

## 종료 기록 형식

```text
[Quality Score]
목표 달성도:
검증 수준:
사용자 개입 감소:
재사용성:
포트폴리오 가치:
운영문서 반영:
총점:
판정:
새 교훈:
규칙 승격 후보:
시스템 개선 후보:
```

## 점수별 자동 개선 행동

| 총점 | 자동 행동 |
|---:|---|
| 25~30 | Golden 후보 여부 확인. 재현 2회 이상이면 `knowledge/goldens.md` 기록 |
| 18~24 | 다음 작업 보완점 1개만 `NEXT.md` 또는 종료 요약에 남김 |
| 10~17 | 실패 원인 1줄 작성 후 `knowledge/failures.md` 후보 검토 |
| 0~9 | 재작업/폐기 판단. `system/continuous_improvement.md` 절차로 개선 후보 필수 작성 |

`시스템 개선 후보`가 있으면 `system/continuous_improvement.md`의 게이트를 통과한 것만 문서에 반영한다.

## 결과물별 추가 기준

### 코드/앱

- 자동 테스트가 있는가
- 실제 런타임 1회 확인했는가
- 외부 부작용이 있으면 명시했는가
- `.env`, API key, 민감정보가 commit 대상에서 제외됐는가

### 콘텐츠/포트폴리오

- `knowledge/copywriting-rules.md` 기준을 적용했는가
- 구린 이유를 1줄로 말할 수 있으면 `quality-bar.md`에 반영했는가
- 결과물이 채용담당자/고객 입장에서 읽히는가

### 리서치/전략

- 출처와 날짜가 있는가
- 미확인 사실을 단정하지 않았는가
- 실행 가능한 다음 1수가 있는가

### 자율작업

- AI가 한 일을 사람이 재현/검증할 수 있는가
- 실제 실행 못 한 항목이 블로커로 분리됐는가
- 보고서가 운영문서에 흡수됐는가

## Golden / Failure 승격

좋은 결과물은 `knowledge/goldens.md`에 다음 형식으로 박제한다.

```text
## YYYY-MM-DD / 프로젝트 / 산출물
왜 좋았나:
재사용 규칙:
링크/파일:
```

실패 결과물은 `knowledge/failures.md`에 다음 형식으로 박제한다.

```text
## YYYY-MM-DD / 프로젝트 / 실패
증상:
원인:
다음 방지 규칙:
승격 대상:
```
