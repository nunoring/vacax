# VacAX Always-On Loop v1

목적: 작업할 때마다 데이터가 남고, 그 데이터가 다음 작업 품질을 바꾸게 만든다. 항상 켜는 것은 기록/평가/교훈 후보 수집이고, 무거운 audit/council/웹조사는 조건부로만 켠다.

## 핵심 루프

```
작업 실행
→ 결과 기록
→ 품질 평가
→ 실패/성공 패턴 추출
→ 시스템 개선 후보 판정
→ 운영 규칙 업데이트
→ 다음 작업 때 자동 적용
→ 이전보다 나아졌는지 테스트
```

## 항상 자동으로 켜는 것

### 분산 담당 채팅 원칙

VacAX는 별도 "Head 채팅" 하나에 의존하지 않는다. 각 담당 채팅이 자기 프로젝트의 작은 Head 역할을 겸한다.

- 담당 채팅은 시작 시 해당 `active/[project]/STATE/NEXT/LOG`를 읽고 Boot한다.
- 담당 채팅은 종료 시 해당 프로젝트의 `STATE/NEXT/LOG/LEARNINGS/EVALS`를 직접 갱신한다.
- `active/vacax_system/`은 상설 지휘 채팅이 아니라 공통 운영 프로토콜 기록 위치다.
- 여러 담당 결과를 합칠 때만 "열린 창 회수" 또는 "시스템 점검"으로 통합 패스를 실행한다.
- UI에서 별도 `VacAX Head` 채팅을 반드시 만들 필요 없다. 이미 열려 있는 VacAX 채팅이 통합 패스를 수행할 수 있다.

### 시작 10초 Boot

세션 시작, "시작", "이어서", 프로젝트명 언급 시 아래만 수행한다.

1. 오늘 작업 프로젝트 1개를 확인한다. 프로젝트가 명확하면 해당 담당으로 바로 시작한다.
2. 해당 프로젝트의 `STATE.md`, `NEXT.md`, `LOG.md` 최근분만 읽는다.
3. `system/stale_check.md` 기준으로 문서 stale 여부를 확인한다.
4. 오늘 목표 1개와 첫 액션 1개를 제안한다.
5. 작업 중 남길 데이터 종류를 미리 고정한다.

출력 형식:

```text
[VacAX Boot]
프로젝트:
문서 상태: fresh / stale / unknown
현재 blocker:
오늘 목표 1개:
첫 액션:
작업 중 기록할 것:
```

### 작업 중 Black Box

작업 중 아래 이벤트가 생기면 종료 때 `LOG.md`/`LEARNINGS.md` 후보로 남긴다.

- 결정 변경
- 막힘/우회
- 실패/반려
- 검증 통과/실패
- 사용자 피드백
- 반복되는 실수
- 새로 발견한 성공 패턴

작업 중에는 전체 문서 갱신을 남발하지 않는다. 후보만 잡고, 종료 때 정리한다.

### 종료 Learning Close

"종료", "끝", "마무리", 작업 완료 시 아래 7개를 닫는다.

1. 결과 기록: `LOG.md` 한 줄
2. 상태 갱신: `STATE.md` 한 화면
3. 다음 액션: `NEXT.md` 3개
4. 품질 평가: `system/quality_scorecard.md` 양식 적용
5. 교훈 후보: 프로젝트 `LEARNINGS.md` 또는 `knowledge/patterns.md` 후보
6. 규칙 승격 필요 여부 판단
7. `system/continuous_improvement.md` 기준으로 시스템 개선 후보 판단
8. 다음 세션 첫 액션 1개 확정

## 매번 켜지 않는 것

아래는 조건부로만 실행한다.

- 전체 프로젝트 audit: 주 1회 또는 "시스템 점검" 요청 시
- 시스템 구조 개선: `system/continuous_improvement.md` 조건 충족 시
- council: 출시/GO-KILL/아키텍처/수익모델/메인 AI 변경 시
- 웹검색: 최신 정책/가격/API/시장 검증 필요 시
- 전체 knowledge 로드: 관련 트리거가 있을 때만
- git commit/push: 종료 루틴과 민감정보 검토 후
- 통합 조정: 프로젝트 간 우선순위 충돌 또는 회수 요청이 있을 때만

## 규칙 승격 기준

```
1회 발생 = LOG에 기록
2회 반복 = LEARNINGS.md 또는 patterns.md 후보
3회 반복 = AGENTS.md/CLAUDE.md 또는 프로젝트 RULES.md 강제 규칙
```

성공 패턴도 같은 기준을 적용한다.

```
1회 성공 = LOG
2회 재현 = goldens.md 후보
3회 재현 = 검증된 패턴(VP)으로 승격
```

## 완료 정의

작업 완료는 "많이 했다"가 아니라 아래를 만족해야 한다.

- 구현 또는 산출물 생성
- 자동검증 또는 사람검증 결과 기록
- 못 한 것과 이유 명시
- 운영문서 갱신
- 다음 1수 확정

코드 작업이면 추가로:

- lint/build/test 중 해당되는 검증 실행
- 실행 못 한 검증은 이유 기록
- 커밋 후보 파일 목록 정리

## 금지

- 기록 없이 새 작업으로 넘어가기
- 별도 Head 채팅이 없다는 이유로 기록/회수를 미루기
- "내일 확인"을 NEXT에 기한 없이 남기기
- 보고서만 쌓고 STATE/NEXT/LOG 미갱신
- 실패 원인을 "느낌"으로 남기기
- 같은 교훈을 3회 이상 반복하고도 규칙으로 승격하지 않기
