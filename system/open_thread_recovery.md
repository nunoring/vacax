# VacAX Open Thread Recovery v1

목적: VacAX 시스템을 켜지 않고 진행한 다른 대화창 3~4개의 작업을 버리지 않고 회수한다. 별도 Head 채팅이 없어도 현재 열려 있는 VacAX 담당 채팅에서 실행할 수 있다.

## 원칙

이미 진행한 대화는 잘못이 아니다. 문제는 그 결과가 `STATE/NEXT/LOG/knowledge`로 흡수되지 않는 것이다. 회수는 "다시 작업"이 아니라 "증거를 운영체제에 연결"하는 일이다.

## 회수 대상

아래 중 하나라도 있으면 회수 대상이다.

- 코드 변경
- 문서/보고서 생성
- 중요한 결정
- 실패/막힘
- 새 교훈
- 다음 액션
- 외부 리서치 결과

## 회수 절차

각 열린 대화창마다 마지막에 아래 요청을 보낸다.

```text
VacAX 회수용으로 이 대화에서 남길 것만 정리해줘.
형식:
1. 프로젝트:
2. 실제로 한 일:
3. 변경/생성된 파일:
4. 검증한 것:
5. 못 한 것/블로커:
6. 새 결정:
7. 새 교훈:
8. 다음 1수:
9. 민감정보/commit 금지 여부:
```

그 답변을 현재 회수 실행 중인 VacAX 세션으로 가져와 아래에 반영한다.

- `active/[project]/LOG.md` 한 줄
- `active/[project]/STATE.md` 현재 상태
- `active/[project]/NEXT.md` 다음 3개
- 필요 시 `DECISIONS.md`
- 필요 시 `knowledge/patterns.md`, `knowledge/failures.md`, `knowledge/goldens.md`

## 우선순위

3~4개 창이 있으면 아래 순서로 회수한다.

1. 코드 변경이 있는 창
2. 외부 배포/게시/결제/계정과 관련된 창
3. 포트폴리오/이직 자료 창
4. 아이디어/리서치 창

## Headless 회수 원칙

- 회수 전용 Head 채팅을 만들 필요 없다.
- 어느 담당 채팅이든 회수 요청을 받으면 해당 프로젝트 문서에 직접 반영한다.
- 여러 프로젝트가 섞인 회수는 각 프로젝트별로 나눠 기록하고, 마지막에 `active/vacax_system/LOG.md`에 통합 요약 1줄만 남긴다.
- 담당 채팅이 자기 작업을 종료 루틴으로 닫았다면 별도 회수는 필요 없다.

## 회수 판단

각 창은 셋 중 하나로 결론 낸다.

- **흡수**: 운영문서와 knowledge에 반영
- **보류**: NEXT에 날짜/조건을 붙여 대기
- **폐기**: LOG에 폐기 이유 1줄만 남김

## 금지

- 모든 대화 전문을 vacax에 복사하지 않는다.
- 민감 개인정보를 git 파일로 옮기지 않는다.
- 회수하면서 새 기능을 더 만들지 않는다.
- Head 채팅을 만들기 전까지 기록을 미루지 않는다.
- 회수 대상이 4개 이상이면 하루에 2개까지만 처리한다.

## 메인 세션 회수 요약 형식

```text
[Open Thread Recovery]
회수한 창:
흡수:
보류:
폐기:
운영문서 갱신:
새 교훈:
다음 1수:
```
