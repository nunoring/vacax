# VacAX 시스템 구조 설명서 — 외부 AI 조언 요청용

> 이 문서를 ChatGPT에 통째로 붙여넣는다.
> 목적: **우리 시스템 구조를 정확히 이해시킨 뒤, 구조 자체에 대한 비판·개선점·사각지대를 받는다.**
> ChatGPT는 외부 컨설턴트 시점에서 본다. (파일·git·Notion 접근은 없다.)

---

## 0. 한 줄 정의

VacAX = 임용후(1인)가 **Claude Code를 메인 오케스트레이터로 두고 AI 멀티에이전트처럼 운영하는 1인 스튜디오**.
- 목적: 4개월 안에 **AX/AI 기획 이직 포트폴리오** 완성
- 제약: 실업급여 4개월, 하루 5~6시간, 자본 최소
- CEO: 임용후 · 공공 B2B 사업기획 2년 2개월 → AI 워크플로우 설계로 전환

## 1. 전체 아키텍처

```
CEO (임용후)
  │
  ├── Claude Code  ← 메인 두뇌 / 오케스트레이터
  │     ├── CLAUDE.md (운영체제: 트리거·게이트·원칙 전부 여기 박힘)
  │     ├── 5직군 (전략가/검증자/개발자/마케터/오케스트레이터)
  │     ├── /council  — 5역할 찬반토론 의사결정
  │     ├── /audit    — 시스템 자동 점검
  │     ├── /watch    — 영상 분석
  │     └── 풀파이프라인 스킬 (시장조사→기획→시뮬레이션→...)
  │
  ├── Notion   — 인사이트 라이브러리(검증패턴/안티패턴/KILL조건) · 프로젝트 인수인계
  └── GitHub   — 운영기록(vacax repo) + 코드(apps repo, 물리적으로 별도)
```

### 작동 방식
- CEO가 자연어로 지시 → Claude Code가 트리거 키워드 감지 → 해당 룰파일 자동 로드 → 직군 모드로 응답
- 모든 결정·상태는 **파일에 기록**. 채팅 기억에 의존하지 않음 (1원칙)
- 코드는 vacax 폴더에 없음. 운영문서와 물리적으로 분리(`Desktop\apps\`)

## 3. 의사결정 시스템 — 5역할 토론 (/council)

중요 결정마다 5역할 찬반토론 → CEO 최종 확정. 최대 3라운드, 설득 가능.

| 역할 | 담당 |
|------|------|
| 🔴 Challenger | 반론·리스크 발굴 |
| 🟢 Advocate | 기회·강점 발굴 |
| 🟡 Realist | 실행 가능성 판단 |
| 🔵 Market | 고객·시장 관점 |
| ⚪ Synthesizer | 수렴·결정안 도출 |

**council 필수 게이트** (CEO 확정 전 반드시 토론): 메인 AI 모델 전환 / 아키텍처 대변경 / 출시 결정 / 수익모델 변경 / 신규 프로젝트 GO·KILL

### 5직군 (Claude 측 사고모드, 각자 거부권 보유)
- **strategist** 시장조사+기획 — 시장 검증 없는 기획안 거부, 알고리즘 의존 수익을 본수익원으로 거부
- **validator** 시뮬레이션+검증 — 통과기준 없는 테스트 거부, 주관적 "잘 됩니다" 보고 금지
- **builder** UX+개발 — 짜기 전 설계 결함 먼저 지적
- **growth** 마케팅+자산화 — 측정 불가 마케팅 금지, 카피 공식 강제
- **operator** 메타+오케스트레이션 (기본)

---

## 4. 풀파이프라인 (신규 프로젝트 처리 흐름)

```
시장조사 → G1게이트(GO/HOLD/KILL) → 기획(PRD) → G2게이트(핵심가치 확정)
  → 시뮬레이션(시나리오 검증) → UX/개발 → 출시
```
각 게이트에서 자동 KILL 조건 체크. 메타 직군이 오케스트레이션.

## 5. 콘텐츠·카피 생산 표준

외부 노출물(포트폴리오·블로그·썸네일·랜딩·README) 전부 자동 적용:
- **6단계 글공식**: 메인후킹→서브후킹→효과입증→전문가포지셔닝→CTA→마무리후킹
- **6상자 카피**: 욕구성/날먹성/충격성/희소성/특정성/정체불명 중 3개 이상
- **제공자 태도**(요청자 어휘 금지) · **모호어 금지**(전부 수치화) · **수단 X 목표 O**

## 6. 안전·운영 원칙 (강제)

1. **AI 기억 불신** — 유일 진실은 파일
2. **운영문서/코드 분리** — 절대 안 합침
3. **API 키 평문 commit 금지** — gitignore + 템플릿만
4. **민감 개인정보 commit 금지** — 개인 신상·금전 등 민감정보는 Notion에만 보관(git 금지)
5. **반대 의견 의무** — CEO 판단 잘못이면 구현 전 push back ("네" 시작 금지)
6. **최소 기록** — STATE 한 화면, NEXT 3줄
7. **세션 시작/종료 자동 루틴** — 시작 시 STATE/NEXT/DECISIONS/LOG 읽고 요약, 종료 시 갱신+commit+Notion 동기화

## 7. 알려진 약점 (자체 진단 — 이게 맞는지도 봐달라)

1. **시각 생성 0** — Claude Code는 이미지·영상 생성 불가. 스토어 스크린샷·피처그래픽·썸네일·홍보영상 전부 외부 의존.
2. **1인 4개월 제약 대비 시스템이 무거울 수 있음** — 5직군·council·풀파이프라인·게이트가 과설계 아닌지.
3. **단일 진실=파일 구조의 확장 한계** — 프로젝트·knowledge 늘면 컨텍스트 관리 부담.

## 8. ChatGPT에게 요청하는 것 (조언 항목)

위 구조를 본 외부 컨설턴트로서 답해달라:

1. **구조적 결함·사각지대** — 이 시스템에서 1인 운영자가 망가뜨리기 쉬운 지점은?
2. **과설계 vs 적정** — 5직군+council+게이트가 4개월 포트폴리오 목표에 비해 과한가, 부족한가? 뺄 것/더할 것?
3. **병목** — 가장 먼저 막힐 지점은? (자체 진단은 '시각 생성'인데 동의하나?)
4. **단일 진실=파일** 방식의 한계와, 더 나은 상태관리 대안이 있나?
5. **이직 포트폴리오 관점** — 채용담당자가 이 시스템 자체를 봤을 때 임팩트 있나, 아니면 산만해 보이나?
6. **우선순위** — 지금 당장 고칠 1가지만 꼽는다면?

각 항목 결론 먼저, 근거 뒤. 막연한 칭찬 금지. 구체적 액션으로.
